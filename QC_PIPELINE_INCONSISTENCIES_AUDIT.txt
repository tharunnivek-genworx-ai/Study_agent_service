1. RETRY ROUTING (retry_routing.py)
================================================================================

1.1 [MED] Redundant validate_section_id_coverage calls
    [FIXED 2026-07-03] structure_coverage_missing_ids() runs once in quality_check_node;
    result passed to structure_check() and classify_retry_routing(structure_missing_ids=...).
    retry_routing reuses the same set for missing_checklist enrichment and
    _structure_coverage_widespread (no second/third validate call on the QC path).
    Standalone classify_retry_routing callers still validate at most once when needed.
    Prior issue: up to three calls per QC pass when det_structure_coverage failed.

1.2 [MED] det_structure_coverage skipped in _map_failures_to_targets
    [FIXED 2026-07-03] Structure coverage resolution and missing_checklist enrichment
    now live in _map_failures_to_targets (via _resolve_structure_missing_ids and
    _enrich_missing_checklist_from_structure). classify_retry_routing passes
    structure_missing_ids through once; mapper returns structure_missing for
    _structure_coverage_widespread. Prior issue: skip in generic loop then duplicate
    block in classify_retry_routing made flow hard to follow.

1.3 [HIGH] Full-regen rule 4 (40% required mc_*) often ineffective for topic_split
    [FIXED 2026-07-03] _affected_required_checklist_ids() now maps failed_section_ids
    (ts_*) to checklist ids via checklist_section_id() before intersecting required.
    Legacy mc_*-as-section-id pipelines still work via union with failed_section_ids.
    Prior bug: affected_required = (failed_section_ids | missing_checklist_ids) & required_mc_ids
    For topic_split pipelines, failed_section_ids are typically ts_* (topic split ids)
    while required checklist ids are mc_*. The intersection was often EMPTY when only
    section-level deterministic failures exist (e.g. Calculus: ts_1-ts_3 failed, all mc_* passed).
    Escalation to full_regeneration via this rule may not fire when intuitively "many
    sections failed" if those sections are ts_* not mc_*.

1.4 [LOW] coverage_ratio = 1.0 when required_ids is empty (vacuous truth)
    [FIXED 2026-07-03] Removed unused coverage_ratio from SectionCoverageResult.
    Pass/fail and routing already use missing_ids only; ratio was never consumed
    in production.

1.5 [MED] LLM full_regeneration recommendation cannot upgrade scoped deterministic mode
    [FIXED 2026-07-03 — INTENTIONAL] Asymmetric reconcile policy is by design.
    _reconcile_mode keeps deterministic section_patch/insert when scoped failures
    map cleanly; LLM retry_recommendation can confirm compatible modes only, not
    escalate to full_regeneration (avoids costly LLM over-escalation; prompt says
    full_regen only for >50% required fail or fundamental teaching misalignment).
    teaching_alignment sole failure still forces full_regeneration when deterministic
    mode is "none" (document-level issue with no section_id patch target).
    Prior audit framed this as a bug; it is a deliberate safety rail — deterministic
    routing corrects unreliable LLM retry_recommendation output in both directions.

1.6 [LOW] unmapped_count threshold (>=3) vs det_* checks
    [FIXED 2026-07-03 — INTENTIONAL] F6 (unmapped_count >= 3) targets routing
    ambiguity from failures that cannot map to a document section_id — mainly LLM
    QC checks with empty or invalid section_id, or must_cover without a resolvable
    checklist id. det_* block-placement checks always carry section_id when the
    section exists, so they increment failed_section_ids (F5 at >= 4) not
    unmapped_count. Prior audit read like a gap; F6 is deliberately LLM/unmappable
    oriented, not a catch-all for deterministic det_* failures.

1.7 [MED] teaching_alignment failures skipped in _map_failures_to_targets
    [FIXED 2026-07-03 — INTENTIONAL] teaching_alignment is a single document-level
    check (whole doc vs teaching instruction) with no section_id patch target.
    _map_failures_to_targets skips it by design; routing uses F1/F2/F7 →
    full_regeneration and qc_feedback, not section_failures / section_patch.
    When bundled with section-level failures, those other checks still drive
    section_patch; alignment is re-evaluated on the next QC pass. Prior audit
    read like a gap; document-level misalignment cannot be surgically patched
    without inventing section blame.

2. FROZEN SETS vs TARGETED MERGE (frozen_sets.py, targeted_merge.py)
================================================================================

2.1 [MED] Frozen must_cover does not protect patched sections on pass 2 [OPEN — BY DESIGN]
    qc_frozen_check_ids omits passed mc_* from FULL QC prompt only.
    On targeted QC (pass 2), targeted_merge EVICTS must_cover for any section in
    reverify_section_ids — even if mc_N was frozen after passing pass 1.
    "Frozen" implies "don't re-check" but patched sections are always re-checked.
    Naming/mental model mismatch. P3/P4 fixes stale *full QC skips* after content
    change; targeted re-verify of patched sections remains intentional.


2.2 [LOW] No frozen list for document_coherence / teaching_alignment [BY DESIGN]
    By design, but means full QC always re-runs document_coherence LLM check and
    all content_accuracy claims. Only must_cover and code get prompt optimization.

2.3 [MED] accumulate_frozen_sets skipped on targeted QC pass
    [FIXED 2026-07-04] refresh_frozen_lineage_after_qc runs on every QC exit (full
    and targeted): prune_frozen_for_sections on targeted, accumulate, rebuild
    qc_section_content_hashes. quality_check_node + test qc_runner both use shared
    helpers (resolve_frozen_for_full_qc / refresh_frozen_lineage_after_qc).

2.4 [HIGH] Frozen ids trusted after content rewrite (regen / patch / resume)
    [FIXED 2026-07-04 — P3/P4] Section content lineage: qc_section_content_hashes
    (JSONB qcsectioncontenthashes) stores SHA-256 baseline per section id at QC
    exit. effective_frozen_sets / resolve_frozen_for_full_qc gates full QC entry;
    study_material_service resume hydration re-validates frozen against active.content.
    NULL legacy hashes → empty effective frozen (safe full QC). Resolves stale-skip
    cases in FULL_REGEN_FROZEN_SETS Cases A,B,C,D,G,H,I,K. See frozen_sets.py.

3. QC FEEDBACK & ARTIFACT LOGGING
================================================================================

3.1 [MED] qc_feedback text block not logged in 05_qc_result.json
    format_qc_feedback() produces qc_feedback in graph state on fail, but
    log_qc_agent payload for qc_result only includes structured qc_result dict.
    The formatted text block is not persisted as its own artifact field.
    Recoverable by re-running format_qc_feedback on logged qc_result.

3.2 [MED] qc_section_failures not logged as separate artifact
    Structured per-section failure bundles (routing.section_failures) live only in
    LangGraph state between QC fail and study_agent retry. Not written to
    05_qc_result.json or a dedicated routing artifact in production (test harness
    writes retry_routing.json).

3.3 [HIGH] Two different feedback channels with no unified audit trail
    section_patch path: qc_section_failures → section_rework_prompt (<sections_to_fix>)
    full_regeneration path: qc_feedback text → <quality_check_feedback>
    Most failures (section_patch) never produce qc_feedback in a way that matters;
    full_regen failures never use structured section_failures in the patch prompt.
    Debugging a retry requires knowing which channel was used.

3.4 [LOW] Reference material duplicated on full_regeneration
    build_qc_feedback_block adds <reference_material_for_correction> in addition to
    <reference_material> already in the main user message. Redundant tokens.

3.5 [LOW] failed_qc_feedback (DB hydration) separate from in-run qc_feedback
    improve/regenerate from previously failed draft uses failed_qc_feedback from DB;
    in-pipeline QC retry uses qc_feedback. Two paths, easy to conflate when reading logs.

4. SECTION PATCH SCOPE & PRESERVATION (section_rework_prompt, document_merge)
================================================================================

4.1 [HIGH] Whole-section replacement, not line-level or field-level patch
    merge_section_patches replaces the entire section dict by id:
      sections[index] = copy.deepcopy(patch)
    No structural merge of subsections, formula_blocks, or content fields.

4.2 [HIGH] No verbatim preservation rules inside patched sections
    improve_prompt and regeneration_prompt have explicit "copy verbatim" rules for
    untouched sections/subsections/blocks. section_rework_prompt does NOT require:
      - Copy subsections not in subsections_to_fix verbatim from current_section_json
      - Volume preservation / anti-thinning for unchanged parts
    Only mandate: fix listed failures, preserve section id.

4.3 [MED] Valid content in OTHER sections is guaranteed; within failed section is not
    Unfailed sections (e.g. ts_4, ts_5 when ts_1-ts_3 fail) are never sent to LLM
    and never touched by merge. Valid subsections/blocks WITHIN a failed section
    rely entirely on LLM instruction-following.

4.4 [MED] current_section_json provided but preservation not enforced
    build_sections_to_fix_block embeds full current_section_json as reference.
    Code does not diff or merge LLM output against it for unchanged parts.

4.5 [LOW] Section patch uses generation sampling when mode=generate
    full_regeneration with generation_mode=generate uses study_generation_temperature
    (0.25), not revision temperature (0.35). Section patch always uses revision
    sampling. Inconsistent retry aggressiveness across retry types.

5. SUBSECTION HANDLING
================================================================================

5.1 [HIGH] Subsections have no ids in document schema
    Only section-level id (ts_*, mc_*). Subsections identified by heading string only.
    No stable subsection key for routing, freezing, or merge.

5.2 [HIGH] Subsection targeting via regex on evidence text only
    _subsection_remediation_targets parses failure evidence with:
      subsection\s+['"]([^'"]+)['"]
    Designed for deterministic _location_evidence format:
      Section 'X', subsection 'Y': detail
    Fragile if evidence wording changes.

5.3 [MED] LLM QC may mention subsections but not in parseable format
    QC schema has section_id only — no subsection_heading field.
    LLM evidence may say "'Definite Integrals' subsection" or
    "The subsection 'Foo' explains..." — regex may or may not match.
    subsection remediation blocks only appear when regex matches.

5.4 [MED] No deterministic-only preservation prompt branch
    No separate prompt section when all failures are det_* block-placement checks.
    Same section_rework_prompt for must_cover depth failures, content_accuracy,
    and det_equation_in_content. Only one bullet names det_equation_in_content.

5.5 [LOW] subsection remediation duplicated in prompt
    subsections_to_fix in <sections_to_fix> JSON plus <subsection_equation_remediation>
    block plus conditional closing text — triple reinforcement, but still soft guarantee.

5.6 [MED] Section-level det_equation_in_content does not set subsections_to_fix
    Calculus example: evidence "Section 'Derivatives': Prose contains display-math"
    → entire section content rewrite implied; no subsection granularity even if
    document has subsections with mixed valid/invalid content.

6. DETERMINISTIC vs LLM CHECK INTEGRATION
================================================================================

6.1 [MED] det_* checks use category document_coherence
    Same category as LLM document_coherence check. Scores bucket section_depth
    bundles must_cover + document_coherence. Deterministic failures affect
    section_depth score alongside LLM must_cover — can be confusing in artifacts.

6.2 [LOW] det_structure_coverage handled differently from other det_* in routing
    [FIXED 2026-07-03 — INTENTIONAL] Skipped in generic mapper loop because it has
    no section_id; resolved via _resolve_structure_missing_ids +
    _enrich_missing_checklist_from_structure inside _map_failures_to_targets (1.2).
    Other det_* use section_id → failed_section_ids. Different target type
    (missing_checklist_ids / section_insert), not a bug.

6.3 [MED] Deterministic failures default to section_patch, not full_regen
    det_equation_in_content on 3 sections → section_patch (threshold is 4+).
    Correct for surgical fix, but combined with 4.1-4.2 means 3 full section
    rewrites rather than minimal edits.

6.4 [LOW] structure_checks vs prose_checks naming in quality_check_node
    block_placement_checks (det_*) grouped under structure_checks variable name
    in node though category is document_coherence not structure.

7. ID MODEL & ROUTING TARGET MISMATCHES
================================================================================

7.1 [MED] ts_* (topic_split) vs mc_* (checklist) dual id system
    topic_split drives section ids (ts_1, ts_2); checklist items have mc_* with
    section_id pointing to ts_*. Routing uses both sets in different rules.
    [FIXED 2026-07-06] Full-regen 40% rule now counts only failed must_cover checklist
    items via failed_must_cover_checklist_ids(); placement-only det_* failures no
    longer map section ids to mc_* for F4. Dual id model remains for other rules.

7.2 [MED] must_cover fail with empty section_id
    Benzene example: mc_4 failed with section_id "" → goes to missing_checklist_ids
    or unmapped depending on resolve path, not failed_section_ids for patch.
    Ambiguous when checklist item exists but section mapping is missing.

7.3 [LOW] fixed_sections cleared by base_qc_return on every QC exit
    [FIXED — INTENTIONAL / DOCUMENTED] QC node always passes fixed_sections=None in
    base_qc_return. study_agent sets fixed_sections; QC reads them on next entry for
    targeted pass. Correct cycle — fixed_sections is study_agent output, not QC output.

8. FULL REGENERATION vs SECTION PATCH
================================================================================

8.1 [MED] Full regen asks for whole document; post-merge may preserve passing sections
    merge_full_regeneration_preserving_passing keeps sections NOT in
    qc_reverify_section_ids from previous_document. LLM asked for full rewrite but
    pipeline may splice old passing sections back — intentional optimization but
    prompt says rewrite everything.

8.2 [LOW] Full regen uses generation_prompt (same as initial generate)
    Not regeneration_prompt. qc_fix_block adds quality_check_feedback + previous_draft.
    previous_failed_qc_block from DB not used in-pipeline.

8.3 [MED] Escalation thresholds may not match user expectation
    3 failed sections → patch; 4+ → full regen. Deterministic math-in-prose on
    3 sections triggers 3 full section LLM rewrites, not minimal line edits.

9. TARGETED QC (PASS 2) CONSISTENCY
================================================================================

9.1 [MED] Patched section re-verifies ALL scoped check types
    targeted_merge evicts must_cover, content_accuracy, document_coherence for
    reverify sections — not just the failure category that triggered the patch.

9.2 [LOW] frozen_check_ids still passed to scoped checklist on targeted verify
    checklist_for_reverify scopes checklist items but frozen ids from pass 1
    don't skip must_cover in targeted pass prompt the same way as full pass.

9.3 [MED] Deterministic structure_checks re-run on targeted pass?
    quality_check_node re-runs block_placement_checks on full merged document
    each QC entry — deterministic checks always fresh on entire doc.
